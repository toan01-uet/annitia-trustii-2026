# AGENTS.md

## Project Overview

- Goal: build machine learning models for the ANNITIA Data Challenge focused on MASLD risk stratification.
- The repository currently contains local challenge data in `data/` and repo guidance in `data-description.md`.
- The required prediction outputs are two patient-level risk scores:
  - `risk_hepatic_event`
  - `risk_death`
- The canonical submission shape is the local hello-world file in `data/hello_world_submission-1773575379610.csv`, with columns:
  - `trustii_id`
  - `risk_hepatic_event`
  - `risk_death`
- Treat this as a patient-level tabular risk-scoring problem. One patient should correspond to one supervised row at modeling time.
- The local dictionary file describes a clean semantic schema, but the raw training CSV may use different physical column names. Do not assume the dictionary names match the raw training table exactly.
- The local raw file currently appears to encode repeated measurements in wide format with visit-style suffixes such as `_v1`, `_v2`, and so on. This is still a tabular dataset. You may engineer summaries from those columns, but do not assume a separate sequence-modeling pipeline unless the user asks for it.
- ONLY STOP/END THE TURN WHEN YOU REACH THE STOP CONDITION SET BY THE USER: As we get deeper into research, it's expected that we will have many iterations without progress in our metric. This is completely normal. Don't get discouraged by it and just look for new angles and unexplored approaches.

## Golden Rules

These override all other guidance. Violating them is never justified.

### Rule 1 -- Start Simple

- Use the smallest viable model for early experiments.
- A data-cleaning or feature-engineering improvement should show up on a simple baseline just as clearly as on a heavier setup, but simple runs are faster and easier to trust.
- Only scale model complexity after the improvement landscape is understood.

### Rule 2 -- Maximize Experiments Per Hour

- Wall-clock time is a scarce resource.
- For every proposed experiment, ask whether the hypothesis can be tested with a cheaper setup first.
- If a change is orthogonal to model architecture, test it on the simplest reliable backbone before combining it with other large changes.
- Avoid entangling multiple changes in one run. Prefer one clear variable per experiment whenever practical.

### Rule 3 -- Trust Your Validation

- Leak-free, consistent cross-validation is a prerequisite, not an experiment.
- Before exploring a new modeling idea, verify that the split strategy, label handling, event-time handling, and score aggregation remain consistent with the official metric.
- Do not introduce postprocessing or validation shortcuts that use ground truth information from the validation fold.
- If a result is faster to obtain but weakens the validity of local validation, reject that shortcut.

## Evaluation And Validation

- Before training serious models, verify the official challenge metric from the local materials or the official challenge page if available.
- If the official metric is not confirmed locally, document the fallback validation metric you use and state clearly that it is a surrogate.
- Default local validation should be patient-level cross-validation with no patient leakage between train and validation folds.
- Prefer 5-fold cross-validation as the default robust setting unless the local data size or target availability makes a different split more defensible.
- If target prevalence or censoring structure matters, use a split strategy that preserves the relevant target balance as much as possible.
- Always report:
  - the split strategy
  - the official metric or surrogate metric
  - a combined score used for model selection
  - separate scores for `risk_hepatic_event` and `risk_death`
  - fold-level scores
- Prefer candidates that improve the combined validation score and show reasonably stable fold behavior.
- Do not rely on a single holdout if repeated cross-validation is feasible.
- Do not use leaderboard feedback as the primary model-selection signal.
- If the problem is fundamentally ranking-oriented, do not force the output into calibrated probabilities unless the challenge metric actually rewards that.
- If event-age columns or post-event information exist in the raw table, never use them as predictors unless the challenge rules explicitly permit that formulation.

## Repository Layout

- `data/`: raw local challenge files. Treat these as immutable source data.
- `data-description.md`: local challenge notes, target framing, and feature-engineering guidance.
- `scripts/`: experiment folders.
- `scripts/expXXX/`: code for one experiment.
- `scripts/expXXX/outputs/`: outputs generated by that experiment.
- There is currently no Python package, no test suite, and no build configuration in this repo. Do not invent extra project structure unless the user asks for it.

## Workflow Expectations

- Start with simple checks before deeper changes:
  - schema inspection
  - raw-to-semantic column mapping
  - duplicate patient IDs
  - null handling
  - target availability
  - train/submission schema alignment
  - row-count sanity checks
- Prefer Polars `LazyFrame` pipelines for feature engineering and joins. Push work into the engine instead of Python loops when practical.
- Use `duckdb` or lazy CSV scans for large-file inspection.
- The local machine has 32 GB RAM and 4 CPU cores. Favor memory-aware transformations, cached intermediates, and reusable feature tables.
- Keep raw inputs untouched.
- Each experiment must live in its own directory using the pattern `scripts/expXXX/`, where `XXX` is the experiment identifier.
- Put experiment code inside that experiment directory.
- Save outputs for that experiment only under `scripts/expXXX/outputs/`.
- Save fold-level validation metrics for that experiment under `scripts/expXXX/outputs/fold_scores.csv`.
- Save the OOF dataframe for that experiment under `scripts/expXXX/outputs/oof_predictions.csv`.
- When inference is requested and `data/test.csv` is available, save prediction artifacts under `scripts/expXXX/outputs/submission/`.
- Build `scripts/expXXX/outputs/submission/submission.csv` against the row order and `trustii_id` values from `data/hello_world_submission-1773575379610.csv`.
- Do not scatter outputs across ad hoc top-level folders such as `artifacts/`, `models/`, `submissions/`, or `tmp` unless the user explicitly asks for a different layout.
- Maintain a root `JOURNAL.md` and append a short entry for each completed run that changes features, params, or ensemble composition.
- After each completed experiment, log the run to Weights & Biases (`wandb`) before moving on.
- Use one `wandb` run per experiment.
- Each `wandb` run should include, at minimum:
  - experiment name
  - timestamp
  - split strategy
  - target setup
  - feature-family summary
  - model/config summary
  - validation metric name
  - combined score
  - per-target scores
  - fold-level scores
  - acceptance or rejection decision
  - paths to any saved validation summaries or artifacts
- Log experiment outputs to `wandb` as well when they exist, such as validation summaries, plots, feature-importance tables, OOF prediction summaries, or submission files explicitly requested by the user.
- When a submission export exists, log both the final `submission.csv` and the raw test-prediction table to `wandb`.
- Rejected experiments must still be logged to `wandb`; do not log only winners.
- If `wandb` credentials or initialization are missing, say so explicitly and treat that as an environment blocker rather than silently skipping tracking.
- Default experiment runs should be validation-only. Do not generate a submission file or dump trained model files for every experiment.
- Accepted experiments should still preserve a clear path from train-time features to test-time inference.
- Only create a full submission file and persist trained model artifacts when the user explicitly asks for them.
- Even for validation-only experiments, save the important outputs under that experiment's `outputs/` directory.

## Modeling Rules

- Avoid target leakage.
- Never use target columns, event occurrence ages, post-outcome measurements, or challenge-only submission values as predictors unless explicitly allowed by the challenge rules.
- Treat `risk_hepatic_event` and `risk_death` as related but distinct targets.
- Shared feature backbones, separate endpoint-specific models, and multi-target strategies are all allowed if validation supports them.
- Prioritize feature engineering over premature model complexity.
- Start from strong, simple baselines before moving to ensembles.
- If the raw table contains repeated visit-style columns, summarize them into leakage-safe tabular features such as:
  - first value
  - last value
  - delta
  - mean
  - min/max
  - standard deviation
  - simple within-row trend summaries
- Prefer clinically meaningful feature groups:
  - burden/comorbidity features
  - ratio and interaction features
  - threshold flags
  - log transforms
  - missingness indicators
  - subgroup-relative features
- Keep a common baseline feature set across both endpoints, then add endpoint-specific blocks when justified.
- If a model outputs arbitrary risk scores rather than probabilities, that is acceptable as long as it aligns with the challenge objective and submission format.

## Code Style Guidelines

- Preserve existing formatting, imports, and file organization.
- Favor small, incremental edits. Do not refactor unrelated code.
- Use descriptive names and readable code.
- Put all library imports at the top of Python files.
- Use `pathlib` for filesystem work, `orjson` for JSON, `httpx` for HTTP, `joblib` for serialization, and `seaborn` for plotting.
- Use timezone-aware datetimes such as `datetime.datetime.now(datetime.timezone.utc)`.
- Prefer Polars and DuckDB for data work. Use pandas only if there is a compelling reason and the user accepts the tradeoff.

## Polars Notes

- Prefer `scan_csv`, lazy joins, lazy aggregations, and parquet caching only when justified.
- Use `.collect_schema().names()` to inspect lazy column names.
- Do not allow object dtypes to leak into lazy plans when reading nested or irregular data.
- For `join_asof`, pass a real `datetime.timedelta` or plain numeric tolerance, not a Polars duration expression.
- Align datetime time units explicitly before as-of joins.

## Security And Data Handling

- Never commit challenge credentials, tokens, cookies, or local secrets.
- Never commit `wandb` API keys, service-account tokens, or local login state.
- Never overwrite the raw CSV files in `data/`.
- Be explicit about the provenance of any external data source, clinical rule, or handcrafted feature file.
- Keep large intermediate files, trained models, notebook outputs, and cached artifacts out of git unless the user explicitly asks otherwise.

## Commit And Review Notes

- Commit after each completed run that changes features, params, or ensemble composition.
- Commit messages should use a concise imperative subject and optional bullet points for important details.
- Do not put email addresses in commit messages.
- Do not use backticks inside `git commit -m` arguments.
- After creating or amending a commit, verify the message with `git show -s --format=%B HEAD`.
- In reviews and summaries, call out leakage risks, validation assumptions, target-definition uncertainty, schema mismatches, and anything not yet verified.
